SBI Grant Tracker API docs

Reference for frontend integration. Most JSON routes use GET; /api/db_migration/reset_tables also accepts POST.

Database selection follows get_db_connection(test_mode=is_test_mode()): set DATABASE_URL for Postgres, or use local SQLite when unset. TEST_MODE controls test DB behavior when not using DATABASE_URL.

User activity routes read session["user_id"]. The client must establish a Flask session with that key before calling them; they do not take user_id as a query parameter.

For mutating actions (bookmarks, checklist, project tasks), the handlers still use GET with query parameters.

Core data endpoints (`api.py`)

GET/api/opportunities

Lists opportunities. Supports either tag search, text search, or default top 50.

Query params
  • tags (optional): comma-separated tags, example education,housing. Matching is case-insensitive.
  • q (optional): free-text search on title/agency (Postgres ILIKE), example tribal health
Precedence
  • If tags is non-empty, tag mode is used (even if q is also present).
  • Else if q is non-empty, search mode is used.
  • Else: simple list of up to 50 rows from grants.
Response shape
  • Default: opportunity_id, title, agency, status
  • With q: same four fields, ordered by posted_date then title
  • With tags: aggregated one object per opportunity with total_score, tag_scores (array of tag / tag_score), plus grant fields selected in SQL (e.g. award_floor, award_ceiling, grant_gov_url). The handler also selects funding_amount; the pipeline schema in init_tables.py currently uses estimated_funding—add a column/alias or change the query so tag mode matches your database.
Example response (`tags=education,housing`)
[
  {
    "opportunity_id": "336489",
    "title": "Community Learning Grant",
    "agency": "Department of Education",
    "status": "posted",
    "total_score": 18.0,
    "tag_scores": [
      { "tag": "education", "tag_score": 10.0 },
      { "tag": "housing", "tag_score": 8.0 }
    ]
  }
]

GET/api/opportunities/total_funding

Aggregated funding from the grants table.

Query params
  • tag (optional): comma-separated tags. When provided, the server filters grants that have matching rows in grant_tags (implementation intended for per-tag totals).
Response shape
  • Without tag: array of rows with agency, total_funding, total_grants (same funding_amount vs estimated_funding caveat as above)
  • With tag: array of per-tag aggregates (depends on DB columns; see handler in app/routes/api.py)

User activity endpoints (`user_activity.py`)

All routes in this section require session["user_id"] to be set. Missing session keys will surface as a server error.

GET/api/user_activity/update_user_grant_status

Creates or updates a user's status for a grant in user_grant_activity.

Query params
  • opportunity_id (required): grant id
  • status (required): arbitrary status string (for example viewed, saved, applied)
Example response
{ "message": "Grant marked successfully" }
GET/api/user_activity/bookmark_grant

Bookmarks a grant for the session user: upserts user_grant_activity with status = 'saved' and is_bookmarked = TRUE (on conflict, only bookmark flag is forced true).

Query params
  • opportunity_id (required)
Example response
{ "message": "Grant bookmarked successfully" }
GET/api/user_activity/unbookmark_grant

Removes the bookmark for the session user on an existing user_grant_activity row: sets is_bookmarked = FALSE and unbookmarked = TRUE so you can query “previously saved then removed” grants separately from never-touched rows.

Query params
  • opportunity_id (required)
Notes
  • Requires a row to already exist for that (user_id, opportunity_id) (typically after a bookmark or status update). No insert is performed here.
  • The user_grant_activity table must include an unbookmarked boolean column for this update and for get_unbookmarked_grants.
Example response
{ "message": "Grant unbookmarked successfully" }
GET/api/user_activity/get_bookmarked_grants

Inserts a task; task_owner_user_id is set from the session user.

Query params
  • project_id (required)

Example response
{ "message": "Task added successfully" }
GET/api/user_activity/get_project_tasks

Lists tasks for a project. Does not require session user in the handler (any client with project_id may call).

Query params
  • project_id (required)
Response

Array of task_id, task_name, task_description, task_owner_user_id, task_project_id.

GET/api/user_activity/update_project_task

Updates a task by task_id.

Query params

  • task_id (required)
  • task_name (required)
  • task_description (required)
  • task_status (required in query; ensure your project_tasks table defines this column if you rely on it)
Example response
{ "message": "Task updated successfully" }

Example response

Maintenance / data reset

GET / POST/api/db_migration/add_unbookmarked_grants

Idempotent schema tweak: adds unbookmarked on user_grant_activity if missing.

GET / POST/api/db_migration/update_grant_gov_url

Backfills grants.grant_gov_url for every row using the same fallback order as ingestion (compute_grant_public_url): use link_url (funding description link) when set; else Grants.gov detail …/search-results-detail/<opportunity_id> when status is posted; else Grants.gov search …/search-grants?keywords=<number> when number is set; else null.

Response
{ "message": "grant_gov_url refreshed …", "rows_updated": 0 }
GET / POST/api/db_migration/reset_tables

Destructive. Drops all tables in the public schema (Postgres) or all user tables (SQLite). Use only in dev or with a full backup.

Response

{ "message": "Tables reset successfully (N dropped)" } on success.

GET/api/reset_oei_data

Deletes grants with opportunity_source = 'wi_psc_oei' and related rows in dependent tables (grant_snapshots, grant_alerts, grant_tags, tribal_eligibility, user_grant_activity, grant_checklist_items).

Response
{
  "message": "WI PSC OEI grants reset successfully",
  "deleted_grant_rows": 0,
  "cleared_opportunity_ids": 0
}

HTML (non-JSON)

The dashboard serves pages under /dashboard, /grants, /alerts, /portal, and this page at /api-docs. Those return HTML, not JSON.

Error format: endpoints generally return HTTP 500 with JSON like { "message": "Error ...: <details>" } when exceptions occur.
